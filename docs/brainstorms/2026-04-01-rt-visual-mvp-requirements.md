# RT Visual MVP Requirements

## Problem Frame

RT 当前已经具备 CLI 形态的核心引擎能力，但使用门槛仍然偏高。单人用户需要手动配置环境变量、理解命令行参数、记忆模型和角色关系，才能发起一场讨论。这使得“配置 Key、选择模型、定义角色、提交任务、观察讨论、查看结果”这条主链路割裂，不利于把 RT 从一个工程原型推进成可直接操作的产品。

本次 MVP 目标是为单人操作者提供一个端到端的可视化工作台，让用户通过界面完成整场会议的配置与运行，并以最小可用的方式看到讨论正在进行到哪个阶段以及最终产出是什么。

## Requirements

**工作台闭环**
- R1. 系统必须提供单人可用的一体化 Web 工作台，覆盖配置、会前编排、会议执行、结果查看四个环节。
- R2. 用户必须能够在不编辑 `.env` 或命令行参数的前提下，通过界面完成一次圆桌会议的启动。
- R3. 系统必须复用现有 RT 引擎能力，保证可视化只是新的操作界面，而不是与 CLI 分叉的另一套执行逻辑。

**配置中心**
- R4. 系统必须提供 API Key 配置界面，允许用户录入、更新和查看各模型提供商的连接状态。
- R4a. API Key 在界面中必须默认遮蔽显示，不要求也不鼓励向用户回显完整明文值。
- R5. 系统必须提供模型配置界面，允许用户启用或禁用可用模型，并设置会议中允许被选择的模型集合。
- R6. Key 和模型配置必须在同一产品流程中可用，用户完成配置后可以直接进入会议编排，而不需要离开当前工作台。

**角色与会议编排**
- R7. 系统必须提供会议创建入口，允许用户填写任务标题、任务描述和项目名，并提交本次会议任务。
- R8. 系统必须支持上传参考附件，作为本次任务的辅助资料输入。
- R8a. 附件上传入口必须复用现有文件验证、Prompt 注入防护和数据分级等安全边界，不能因为切换到 Web UI 而绕过现有保护。
- R9. 系统必须提供内置角色模板，并允许用户在发起会议前微调角色名称、职责说明、角色指令和绑定模型。
- R10. 用户必须能够通过界面决定本次会议启用哪些角色，从而形成实际参与讨论的角色集合。
- R10a. 会议启动时，系统必须固化本次会议实际使用的角色、模型、任务输入和附件清单快照，用于结果页回看与追溯。

**过程可视化**
- R11. 系统必须提供会议执行页，并以阶段级视图展示会议当前进展。
- R12. 阶段视图必须至少展示当前阶段、已完成阶段、执行状态以及每个阶段的摘要结果。
- R13. MVP 不要求展示角色逐条发言、推理过程、原始 prompt、检索细节或底层调试信息。
- R13a. 当会议执行失败或中断时，阶段视图必须向用户展示失败发生在哪个阶段、失败原因摘要，以及可执行的下一步动作。

**结果查看**
- R14. 会议完成后，系统必须提供结果页展示最终报告与阶段摘要。
- R15. 结果页必须支持回看本次会议所使用的模型配置、角色配置以及上传附件清单。
- R16. 结果页优先满足“可读、可回看、可追溯本次配置”，不要求在 MVP 中支持结果后二次编辑再运行。

## Success Criteria

- 单人用户可以在一个界面中完成 Key 配置、模型启用、角色微调、任务创建、附件上传、会议启动和结果查看。
- 用户无需接触命令行即可完成一次完整会议。
- 会议执行过程中，用户能明确知道当前处于哪个阶段，以及阶段是否成功完成。
- 会议结束后，用户能清楚回看“这次会议是怎么配出来的”，包括角色、模型和附件。
- Web UI 不会削弱 CLI 阶段已完成的安全边界，尤其是附件校验与输入防护。

## Scope Boundaries

- 仅面向单人操作者，不包含团队共享、成员协作、权限控制或管理员分层。
- 仅实现 MVP 级过程可视化，不展示角色级发言流、模型重试、降级、checkpoint 明细或原始调试数据。
- 角色来源于内置模板的微调，不在 MVP 中支持从零创建任意新角色体系。
- 仅要求支持通过界面上传任务附件，不要求在 MVP 中定义复杂的资料治理工作流。
- 不要求在 MVP 中支持会议完成后的“修改配置并再次运行”闭环。

## Key Decisions

- 单人工作台优先：先把个人可用闭环做通，再考虑团队与权限模型，避免过早引入后台管理复杂度。
- 端到端一体化优先：MVP 不拆成独立配置工具和独立会议工具，而是保持单一工作流，减少用户切换成本。
- 角色采用“模板可微调”：在灵活性和复杂度之间取中间值，既满足不同会议场景，也避免把角色系统做成完整提示词编辑平台。
- 过程可视化限定为阶段级：优先解决“看得见流程正在推进”，而不是一开始做成调试控制台。
- 结果页以可追溯为核心：MVP 的结果不只看结论，还要能回看会议所依据的角色、模型和资料。

## Dependencies / Assumptions

- 现有 CLI 引擎具备稳定的会议执行能力，并可以被 Web 层复用。
- 现有附件处理、模型调用、角色执行与最终报告生成逻辑可以被接口化，而不是只能从命令行触发。
- 用户接受先通过单机或本地部署方式使用 MVP，而不是一开始追求多用户在线服务。
- Web UI 的推进顺序需要在规划阶段明确：是先基于 mock 或本地假数据完成界面闭环，还是以打通真实 API 调用为前置条件。
